kafka爬坑-docker中部署

发表于 2019-07-29 | 更新于 2019-08-19 | 分类于 中间件 | 评论数：

如何在docker中部署kafka。然后关于listener的配置

主要是为了在本地搭建一个开发环境。然后用了docker-compose来管理。发觉能够连上。但是发消息的时候一直报org.apache.kafka.common.errors.TimeoutException: Timed out waiting for a node assignment.
然后翻到了stackoverflow-Robin Moffatt,然后他推荐阅读这边关于listener的文章，顿时感觉明白了问题所在。

有问题的配置

然后能工作的就是下面一个配置

首先关于几个配置。

  • KAFKA_LISTENERS： kafka监听外部的端口。
  • KAFKA_ADVERTISED_LISTENERS: kafka建议client去监听的端口。
  • KAFKA_LISTENER_SECURITY_PROTOCOL_MAP：protocol的mapping

下面几个概念，是我花费很多时间搞懂的概念。

  • kafka本身就是为了分布式考虑的。这也就是advertiesd_listeners存在的意义
    • 举个例子，如果你配置的是broker A。但是A不是leader。而B是，那么这里advertised_listener最好配置B
    • 他们需要内部通信。即使只有一个broker的standalone的模式，他的解决方案是自己和自己说话。
  • client端和broker第一次握手的之后，则是会获得一些元些信息。
    • 其中之一就是，会获得所有partition的信息。然后去连其他的broker。而这里就是advertiesd_listeners
  • LISTENERS是可以配置多个的。

所以，当client和broker，没有办法同时路由到kafka这个hostname中，就会造成这个问题。 - 如果client无法路由到kafka，比方说宿主的hostname是NotKafka，那么他能够连上broker。但是他发消息，因为他发消息是发给ADVERTISED_LISTENERS - 如果client无法路由到kafka，那么也就起不来，因为他无法获得监听端口 - 如果ADVERTISED_LISTENERS不一致，也起不来，你看后台，他会报一直在选举leader

那么解决方案就是，保证两者同时能够路由到网址。我想到三个方案

  1. 在client端的hosts或者其他方式加入路由，强制client端路由到
  2. 就是我的方案。设立两个监听端口。然后KAFKA_INTER_BROKER_LISTENER_NAME来指定内部用哪一个。
    • 内部用的，最好用PLAINTEXT或者其他系统默认的。因为我在用confluentinc的schema-schema-registry中，发觉他会检查，并且写死。
  3. 强制broker能够路由到自己。
    • 我记得docker-compose里面可以配置多个network之类的。但是我具体忘记了。
