kafka 配置

发表于 2019-03-08 | 分类于 中间件 | 评论数：

kafka的基础配置。主要是从《kafka权威指南》的笔记

思维导图

思维导图

常规配置

  1. broker.id: 集群唯一
  2. port： 端口
  3. zookeeper.connect: zookeeper地址
  • 格式是 hostname:port/path
  • 多组用分号隔开
  1. log.dirs: 消息的磁盘位置。后面可以看到，消息基本都是以log。
  • 多组可以用逗号隔开。
  1. num.recovery.thread.per.data.dir:可以理解为启动的那一次的并发线程数。可以大一点，因为防止重启。左后用的数据个数，是这个数值乘以目录树。以下为几个时间点。
    • 正常启动， 用于打开每个分区的日志片段
    • 奔溃重启后，检查和截短每个分区
    • 关闭时候，关闭日志片段。
  2. auto.create.topic.enable:自动创建主题。可能有以下几个时间点。
    • 当一个生产者开始写时候。
    • 当一个消费者开始读取时候
    • 客户端向主题发送元数据请求时。

主题默认配置

  1. num.partitions: 新主题包含多少分区。
    • 分区可以张家，不能减少

  2. log.retention.ms/log.retention.hours/log.retention.minutes: 主题保留的时间。如果有多个被设置。那么具体数值最小的参数。

  3. log.retention.bytes: 分区保存的最大字节数。主题就是所有分区的和。

  4. log.segment.bytes: 这里是日志片段。不是太清楚，感觉像是borker上物理的文件大小。
    • 如果消息到borker上，就被写入segment
    • segment满了，就停止写入。等待过期之后释放。

  5. log。segment.ms: 多久被关闭。

  6. message.max。bytes：消息的大小限制

总结

  • segment关闭，只是不再写入？
  • 当有几个配置。比如log。segment.ms和log.segment.byte同时配置，则是看先触发哪一个。

硬件

  1. 磁盘影响生产者，内存影响消费者
  2. AWS。建议用M4和r3。
    • M4允许较长的保留数据
    • R3有较高的吞吐量
    • 对于AWS不是太熟，貌似现在这些都升级了。
  3. java GC: 其实具体配置可能比较考验水平。但是作者说，kafka内存使用率非常高。容易产生对象，需要早点启动。
