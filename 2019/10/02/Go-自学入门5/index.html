Go-自学入门5

发表于 2019-10-02 | 分类于 Go | 评论数：

Go 自学笔记。主要还是在读《Go语言实战》的第六章的笔记主要是关于类型的

个人感想

  1. Go的关于并发理念其实是一种经验总结，把以前很多经验，给整合到语言层面去了。其实这也是Go一直秉持的理念
  2. 并发和并行。书中解释的很完美，其实读了之后，有种醍醐灌顶的感觉，说真的，以前一种把这两个概念给混淆。不过这个概念清楚了之后，很多东西其实反而豁然开朗
  3. channel其实就是一个简单的MQ。不过功能很少

第一天

  • Go语言的并发同步模型来自于一个叫做通信顺序进程(Communicating Sequential Processes,CSP)的泛型。
  • 逻辑处理器和goroutine：类似于netty的机制。逻辑处理器可以理解为主线程。而goroutine，其实是一个个逻辑单元。由逻辑处理来决定，goroutine是否能够被执行以及和线程调度的相关内容。
    • goroutine:是一个独立的运算单元
    • 当一个goroutine抢占阻塞资源，而被阻塞时候，这个线程会被挂起，然后一条新的线程回合逻辑处理器挂钩。
    • 如果是网络，则是会进入特别的网络轮询器。不知道会不会产生自己新的轮询器？
    • 这个叫做分离
    • 最多创建10000个线程，通过runtime/debug包的SetMaxTreads来更改
    • 逻辑处理器，类似于CPU，如果一个goroutine运行时间过长，则是会提前退出的。
  • 并发（concurrency）不是并行（parallelisim）。这个是反复强调的事情。其实这也是Go的理念。
    • 并行关键是同时做很多事情
    • 并发是同时管理很多事情。
    • 其实我的理解是，并行关键是事情快点做，类似于快速响应，而并发的关键在于事情要做好。程序不要爆掉。可以放在后面做。
    • 书中的图6-2和图6-3说的比较详细，但是这里没有办法显示出来。
  • defer这个关键词，好像是挺有用的，延时加载
    • 多个defer的时候，顺序是倒过来的。
  • runtime.GOMAXPROCS(1) 设定逻辑处理器的最大值
  • sync.WaitGroup java里面的CountdownLatch

第二天

  • runtime.Gosched: goroutine退出现在线程。等待下次被调用。
  • go build –race: 检查程序中有没有一些线程问题。
  • 可以在头上这样定义一下文件的变量
  • 共享资源
    • atomic.AddInt64(&atomicCount, 1): 这个不是互斥，而是在Go中，共享变量，是其他goroutine copy到自己，然后就是利用这个，不停的去同步。
    • mutex *sync.Mutex，互斥锁。用法和final一样。
  • channel:本质上来说，就是一个MQ
    • 基本操作
    • 无缓冲: 接受和发送都会被阻塞
    • 有缓冲：只有接受在没值的时候会被阻塞。
      • 关闭后，只能取，不能塞
      • 已关闭，没数据的channel，会返回该通道的零值。如果加入可选标志，还能得到通道信息。就是value,ok :=<=channel中的ok值会是false。这样判断比较好。
      • 本身不支持timeout。这里是支持timeout的官方方法: 用select分支来做
      • 这里说明了一个问题： 就是timeout的routinue时间太长。timeout的期间的goroutinue不停的进入逻辑处理器。造成问题。而这个，就是等到返回值之后，终止了goroutine
  • 处理错误的三种方法: 简单的来说，就是创建一个对象。做为第二参数扔回去。
